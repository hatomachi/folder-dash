# フォルダダッシュ (folder-dash) アーキテクチャガイド

本ドキュメントは、本プラグインの機能改修やエンハンス開発を行う開発者・メンテナー向けの設計・実装方針資料です。

## システムアーキテクチャの基本方針
- **Gitでの差分管理を最優先したステートレス設計**: プラグイン内部に隠しJSONデータベースや特殊な永続化ストレージを持たず、**すべての状態データ（メトリクス、履歴、担当者）は `_Task.md` および `_Epic.md` のYAMLフロントマター（プロパティ）内にプレーンテキストとして保持**します。これにより、別環境での作業やチームメンバー間でのタスク状態の共有が、通常のGitフローに自然と乗る堅牢な設計となっています。
- **カスタム・アイテムビュー (Left Pane ItemView)**: v2.0.0より、メインのノート編集領域を妨げないよう、Obsidian の `ItemView` API を活用し、専用の左ペイン（サイドバー）にダッシュボードを常駐・描画するアーキテクチャへ移行しました。メインペインのファイル遷移に合わせて動的にUIを構築します。
- **バックログボード (Main Pane View)**: v2.1.0より、プロジェクト全体を俯瞰するメインペイン用ダッシュボード (`VIEW_TYPE_BACKLOG_BOARD`) を導入。メタデータキャッシュを利用し、フォルダツリー全体からタスクとEpicを走査して動的に一覧化します。

## 担当者（ユーザー）の自動取得の仕組み
チームメンバーに「プラグインの設定画面から自分の名前を入力する」という手間を掛けさせないため、各個人のローカルGit設定環境を利用しています。
- Obsidian は Electron（Node.js）環境上で動作するという特性を利用し、`child_process` モジュールの `exec` を呼び出して `git config user.name` コマンドを非同期実行しています。
- `app.vault.adapter` 経由で取得した Vault（プロジェクト）のルート絶対パスをカレントディレクトリ（`cwd`）として参照するため、グローバル設定だけでなく当該リポジトリ固有の config 設定にも正確に対応します。
- 取得エラー時やGit未設定環境下では例外をキャッチし、`'Unknown User'` へフォールバックする安全な作りとしています。

## データ構造とマーカーファイル (v2.1.0~)
データはフォルダツリー内に配置される特殊な名前のマークダウンファイル（マーカーファイル）によりメタデータ管理されます。主に以下の構造を前提としています。

### タスクマーカー (`_Task.md` / 以前の `_Summary.md`)
各タスクフォルダ直下に配置され、タスクの状態を管理します。
```yaml
title: string                # タスク名
assignee: string             # 現在の主担当者（自動取得されたGit名）。アクションを起こしたユーザー名でテイクオーバー（上書き更新）される。
do_today: boolean            # 「今日やる」タスクかどうかのフラグ（「🌟」ボタン押下時にトグル更新され、フィルターとして利用される）
status: string               # 現在の状態（'not-started', 'in-progress', 'blocked', 'completed'）
created_at: string           # タスクノート生成時のISO日時文字列
started_at: string           # 最初に「着手」ボタンが押されたISO日時文字列
completed_at: string         # 最後に「完了」ボタンが押されたISO日時文字列
work_time_minutes: number    # 着手状態（in-progress）であった累積時間（分単位）
block_time_minutes: number   # ブロック状態（blocked）であった累積時間（分単位）
last_toggled_at: string      # 最後にステータス遷移が起きたISO日時文字列（累積時間計算の中間状態として利用）
latest_update: string        # 会議等で確認する最新状況のハイライト（リッチ編集可能）
history: Array<{             # アンドン・テイクオーバーの作業履歴（タイムライン）
  time: string,              # アクション発生のISO日時
  action: string,            # 'start' | 'block' | 'complete'
  user: string,              # そのアクションを実行したGitユーザー名
  reason?: string            # 'block' アクション時にModalから入力された待機理由
}>
```

### エピックマーカー (`_Epic.md`)
タスクを束ねる上位テーマ（Epic）のフォルダ直下に配置され、バックログボードでのグルーピング判定キーとして使われます。
プラグインはタスクフォルダからVaultルートに向けてディレクトリツリーを再帰遡上し、最も近い `_Epic.md` (`EPIC_MARKER_FILE`) を持つフォルダ名をそのタスクの「所属エピック名」、同パスを「所属エピックパス」として採用します（実装メソッド: `getEpicInfoForTask` ）。
```yaml
title: string                # エピック名
status: string               # エピック全体でのステータス
created_at: string           # 作成日時
```

## UIと状態の分離に関する工夫
### `processFrontMatter` による安全な状態更新
ダッシュボード上のボタン（UI側）からの状態変更リクエストは、すべて Obsidian 標準 API の `app.fileManager.processFrontMatter(file, callback)` を通じてキューへ渡されます。
このメソッドを利用することで、フロントマターのパース、オブジェクト変更の適用、そして再シリアライズがスレッドセーフに行われます。ユーザーがエディタ上で同じファイルのメタデータをマニュアル編集している最中であっても、データの競合（コンフリクト）や破損リスクを最小化しています。
Agendaビューにおける `latest_update` モーダルからのリッチテキスト（改行やカラータグ等のインラインHTML）の保存も、このAPIを通して安全にメタデータに注入され、エディタ内でエスケープさせずにネイティブHTML（`innerHTML`）として描画されます。

### 自動再描画のイベントリスナー設計
ダッシュボードとノートの閲覧・編集を完全に分離し、変更監視を行っています。
- `app.workspace.on('file-open')` イベントリスナーを使用し、左ペインダッシュボードでは現在メインペインでアクティブなファイルが属するフォルダを常時監視しています。
- メインペインのバックログボードや左ペインは完全にキャッシュ駆動となっており、`app.metadataCache.on('changed', ...)` を監視しています。タスクマーカー（`_Task.md`）のプロパティ変更が検知された瞬間にフロントエンド（ボード）を自動で `renderBoard()` 等により再描画します。これにより、別ペインでの手動書き換えやGit Pull等による外部からのファイル変更発生時も、常にUIが最新のYAML状態と同期されるシームレスな仕組みを実現しています。
